# Веб-платформа для совместного планирования путешествий «Friends Journey»

**Пояснительная записка к выпускной квалификационной работе**

---

## Аннотация

В рамках работы спроектирована и реализована веб-платформа «Friends Journey» для совместного планирования путешествий небольшими группами друзей (2–6 человек). Система объединяет в едином рабочем пространстве поездки: совместное расписание по дням, интерактивную карту с точками маршрута, групповые голосования, учёт и автоматическое разделение расходов, групповой чат и встроенного ИИ-ассистента, который анализирует переписку и отвечает на вопросы участников с учётом контекста поездки.

Приложение построено как полнофункциональное full-stack решение: одностраничное клиентское приложение (SPA) на React/TypeScript и серверный REST API на NestJS/TypeScript с реляционной базой данных PostgreSQL. Интеграция искусственного интеллекта реализована через API Anthropic Claude с использованием механизма принудительного вызова инструментов (tool use) для получения строго структурированного ответа.

**Ключевые слова:** веб-приложение, SPA, React, NestJS, PostgreSQL, Prisma, REST API, JWT, искусственный интеллект, Claude, Mapbox, планирование путешествий, совместная работа.

## 1. Введение

### 1.1. Актуальность

Самостоятельные групповые путешествия стали массовым форматом отдыха. При этом организация поездки группой из нескольких человек сопряжена с типовыми трудностями: обсуждение «расплывается» в мессенджере, договорённости теряются в потоке сообщений, выбор мест превращается в бесконечную переписку, а учёт общих расходов и взаиморасчётов ведётся вручную и порождает конфликты. Существующие инструменты решают эти задачи фрагментарно: карты — отдельно, разделение счетов — отдельно, обсуждение — в стороннем мессенджере.

Актуальность работы состоит в создании единого пространства, в котором группа планирует поездку целиком: от карты и расписания до бюджета и согласования спорных вопросов через голосования, — c подключением ИИ-ассистента, снимающего рутину анализа переписки и подбора рекомендаций.

### 1.2. Цель работы

Цель — спроектировать и реализовать веб-платформу для совместного планирования путешествий, обеспечивающую совместную работу группы над маршрутом, расписанием, бюджетом и решениями, с интеграцией ИИ-ассистента.

### 1.3. Задачи

1. Проанализировать предметную область и сформулировать функциональные требования.
2. Спроектировать архитектуру клиент-серверного приложения и модель данных.
3. Реализовать серверный REST API с аутентификацией и разграничением доступа.
4. Реализовать клиентское SPA с полным набором экранов продукта.
5. Реализовать функциональные модули: поездки, участники, расписание, голосования, бюджет, чат.
6. Интегрировать картографический сервис для отображения точек маршрута.
7. Интегрировать ИИ-ассистента для ответов в чате и анализа переписки.
8. Провести проверку корректности работы системы.

### 1.4. Объект и предмет исследования

**Объект** — процесс совместного планирования группового путешествия.
**Предмет** — программные средства и архитектурные решения для веб-платформы, поддерживающей этот процесс.

### 1.5. Методы

Использованы методы объектно-ориентированного и компонентного проектирования, реляционного моделирования данных, проектирования REST API, а также практики современной веб-разработки (типизация, декларативное управление состоянием, миграции схемы БД).

## 2. Анализ предметной области

### 2.1. Сценарии использования

Типовой сценарий: один из участников создаёт поездку, приглашает друзей, после чего группа совместно наполняет её содержанием — добавляет дни и пункты расписания, отмечает места на карте, обсуждает детали в чате, выносит спорные вопросы на голосование и фиксирует общие расходы с автоматическим разделением сумм. ИИ-ассистент помогает: отвечает на вопросы о направлении прямо в чате и собирает «сводку договорённостей» из переписки.

### 2.2. Обзор аналогов

| Категория инструментов | Примеры | Ограничения |
|---|---|---|
| Карты и маршруты | Google Maps, Google My Maps | Нет общего бюджета, голосований, чата |
| Разделение расходов | Splitwise, Tricount | Только финансы, нет маршрута и расписания |
| Заметки/доски | Notion, Google Docs | Нет карты, нет автоматики по расходам |
| Мессенджеры | Telegram, WhatsApp | Договорённости теряются, нет структуры |

Вывод: на рынке отсутствует интегрированное решение, объединяющее карту, расписание, бюджет, голосования и обсуждение в одном продукте с ИИ-поддержкой. Это определяет нишу разрабатываемой системы.

### 2.3. Целевая аудитория

Небольшие группы (2–6 человек), путешествующие самостоятельно, ценящие свободу планирования и совместное принятие решений.

## 3. Постановка задачи и требования

### 3.1. Функциональные требования

- **F1. Аутентификация:** регистрация, вход, выход; защита приватных разделов.
- **F2. Поездки:** создание, просмотр списка, открытие, редактирование, удаление; разграничение доступа по участию.
- **F3. Участники:** приглашение по email, отображение состава, удаление участника организатором.
- **F4. Расписание:** дни поездки и пункты внутри дня (тип, время, заголовок, заметка, координаты).
- **F5. Карта:** отображение точек маршрута, сгруппированных по дням, на интерактивной карте; поиск места по названию.
- **F6. Голосования:** создание голосования с вариантами, подача/смена/отмена голоса, подсчёт результатов.
- **F7. Бюджет:** учёт расходов с категориями, автоматическое разделение суммы между выбранными участниками, расчёт баланса каждого.
- **F8. Чат:** обмен сообщениями с сохранением истории; обращение к ИИ-ассистенту через упоминание `@гид`.
- **F9. ИИ-анализ:** формирование структурированной «сводки договорённостей» по переписке.

### 3.2. Нефункциональные требования

- **N1.** Типобезопасность кода (TypeScript на клиенте и сервере).
- **N2.** Валидация входных данных на сервере.
- **N3.** Хранение паролей в виде хешей; передача сессии в защищённой cookie.
- **N4.** Разграничение доступа: операции над поездкой доступны только её участникам.
- **N5.** Адаптивный, единообразный пользовательский интерфейс.
- **N6.** Поддержка двух языков интерфейса (русский/английский).

## 4. Архитектура системы

### 4.1. Общая схема

Система построена по клиент-серверной архитектуре. Клиент (SPA) общается с сервером по REST API поверх HTTP; сессия передаётся в HttpOnly-cookie. Сервер реализует бизнес-логику и обращается к PostgreSQL через ORM Prisma. Для внешних сервисов сервер обращается к API Anthropic (ИИ), а клиент — к API Mapbox (карта и геокодирование).

```
┌───────────────┐     HTTPS/REST (cookie)   ┌────────────────────┐
│   Браузер     │ ───────────────────────►  │   NestJS API       │
│  React SPA    │ ◄───────────────────────  │  (REST, бизнес-    │
│  (Vite)       │        JSON               │   логика, guard'ы) │
└──────┬────────┘                           └─────────┬──────────┘
       │ Mapbox GL JS / Geocoding                     │ Prisma ORM
       ▼                                              ▼
┌───────────────┐                           ┌────────────────────┐
│  Mapbox API   │                           │   PostgreSQL       │
└───────────────┘                           └────────────────────┘
                                                      │
                                            HTTPS     ▼
                                            ┌────────────────────┐
                                            │ Anthropic Claude   │
                                            │ API (ИИ-ассистент) │
                                            └────────────────────┘
```

### 4.2. Организация репозитория (монорепозиторий)

Проект организован как монорепозиторий на основе npm workspaces, что упрощает совместное использование зависимостей и единый процесс сборки.

```
friends-journey-platform/
├── apps/
│   ├── web/        # Клиент: Vite + React + TypeScript
│   └── api/        # Сервер: NestJS + Prisma
├── packages/
│   └── shared/     # Общие типы (зарезервировано)
├── package.json    # Корневые workspaces
└── ДИПЛОМ.md       # Настоящий документ
```

### 4.3. Архитектура клиента

Клиент следует принципам Feature-Sliced Design — разделение по слоям ответственности:

- `app/` — корневой компонент, провайдеры, маршрутизация, глобальные стили;
- `pages/` — страницы (лендинг, вход, регистрация, дашборд, страница поездки);
- `features/` — функциональные блоки (аутентификация, дашборд, вкладки поездки);
- `shared/` — переиспользуемый код: `api/` (клиент к серверу), `ui/` (компоненты), `lib/` (утилиты, контексты, i18n).

### 4.4. Архитектура сервера

Сервер построен на модульной архитектуре NestJS. Каждая предметная область — отдельный модуль с тремя слоями: контроллер (HTTP-маршруты), сервис (бизнес-логика), DTO (валидация входных данных). Доступ к БД инкапсулирован в глобальном `PrismaModule`.

Модули: `AuthModule`, `UsersModule`, `TripsModule`, `ItineraryModule`, `VotesModule`, `ExpensesModule`, `ChatModule`, `AiModule`, `PrismaModule`.

### 5.3. Обоснование ключевых решений

- **Единый язык (TypeScript) на клиенте и сервере** сокращает контекстные переключения и позволяет переиспользовать типы.

- **JWT в HttpOnly-cookie** выбран как более защищённый от XSS вариант по сравнению с хранением токена в localStorage.
- **Claude Haiku** выбран как быстрая и экономичная модель, хорошо работающая с русским языком; механизм tool use гарантирует машиночитаемый ответ.

## 6. Модель данных

### 6.1. Перечень сущностей

База данных включает 17 таблиц. Управление схемой ведётся декларативно (Prisma schema) с версионированием через миграции.

| Сущность | Назначение |
|---|---|
| `User` | Пользователь (имя, email, хеш пароля, аватар) |
| `Trip` | Поездка (название, описание, направление, даты, статус, владелец) |
| `TripMember` | Связь «пользователь ↔ поездка» с ролью |
| `TripDay` | День поездки (дата, порядковый номер) |
| `ScheduleItem` | Пункт расписания (тип, заголовок, время, заметка, координаты, адрес) |
| `TripRoute`, `RoutePoint` | Маршруты и их точки (зарезервировано под расширение) |
| `PlaceCache` | Кэш мест (POI) из внешних провайдеров (зарезервировано) |
| `TripMessage` | Сообщение чата (текст, флаг бота, автор) |
| `ChatSummary` | Сжатая «память» блока чата (настроение, сводка, темы, решения, вопросы) |
| `TripChat` | Метаданные импортируемого внешнего чата (зарезервировано) |
| `ChatAnalysis` | Результат ИИ-анализа переписки (сводка + JSON-секции) |
| `Vote`, `VoteOption`, `VoteAnswer` | Голосование, его варианты и голоса участников |
| `Expense`, `ExpenseSplit` | Расход и его доли по участникам |

### 6.2. Ключевые связи

- `Trip` 1—∞ `TripMember` ∞—1 `User` (состав поездки; ограничение уникальности пары `tripId+userId`).
- `Trip` 1—∞ `TripDay` 1—∞ `ScheduleItem` (расписание по дням).
- `Trip` 1—∞ `Vote` 1—∞ `VoteOption` 1—∞ `VoteAnswer` (голос пользователя уникален по паре `optionId+userId`).
- `Trip` 1—∞ `Expense` 1—∞ `ExpenseSplit` (доли расхода по участникам).
- `Trip` 1—∞ `TripMessage` (история чата; индекс по `tripId+createdAt`).
- `Trip` 1—∞ `ChatAnalysis` (история запусков анализа).

Для дочерних сущностей настроено каскадное удаление (`onDelete: Cascade`): удаление поездки удаляет связанные дни, голосования, расходы, сообщения и т. д.

### 6.3. Замечания по проектированию

- Идентификаторы — UUID, что упрощает распределённую генерацию и не раскрывает порядок создания записей.
- Денормализация отсутствует; агрегаты (баланс участника, итоги по категориям) вычисляются на лету.
- Поля `lat`/`lng`/`address` в `ScheduleItem` хранят геокоординаты пункта для отображения на карте.
- В `ChatAnalysis` хранится как разобранная сводка (`summary`), так и исходный структурированный ответ модели (`rawModelOutput`) — для прослеживаемости.

## 7. Проектирование REST API

Все ответы — в формате JSON. Приватные маршруты защищены guard'ом, проверяющим JWT из cookie. Глобальный префикс — `/api`.

### 7.1. Аутентификация и пользователь

| Метод | Путь | Назначение |
|---|---|---|
| POST | `/api/auth/register` | Регистрация, установка сессионной cookie |
| POST | `/api/auth/login` | Вход, установка cookie |
| POST | `/api/auth/logout` | Выход, очистка cookie |
| GET | `/api/users/me` | Текущий пользователь (защищено) |

### 7.2. Поездки и участники

| Метод | Путь | Назначение |
|---|---|---|
| GET | `/api/trips` | Список поездок пользователя |
| POST | `/api/trips` | Создать поездку (автор становится участником-владельцем) |
| GET | `/api/trips/:id` | Поездка по идентификатору (только участник) |
| PATCH | `/api/trips/:id` | Изменить поездку (только владелец) |
| DELETE | `/api/trips/:id` | Удалить поездку (только владелец) |
| POST | `/api/trips/:id/members` | Пригласить участника по email |
| DELETE | `/api/trips/:id/members/:memberId` | Удалить участника (только владелец) |

### 7.3. Расписание

| Метод | Путь | Назначение |
|---|---|---|
| GET | `/api/trips/:tripId/days` | Дни с пунктами |
| POST | `/api/trips/:tripId/days` | Добавить день |
| PATCH | `/api/trips/:tripId/days/:dayId` | Изменить день |
| DELETE | `/api/trips/:tripId/days/:dayId` | Удалить день |
| POST | `/api/trips/:tripId/days/:dayId/items` | Добавить пункт |
| PATCH | `/api/trips/:tripId/days/:dayId/items/:itemId` | Изменить пункт |
| DELETE | `/api/trips/:tripId/days/:dayId/items/:itemId` | Удалить пункт |

### 7.4. Голосования

| Метод | Путь | Назначение |
|---|---|---|
| GET | `/api/trips/:tripId/votes` | Список голосований с результатами и голосом пользователя |
| POST | `/api/trips/:tripId/votes` | Создать голосование |
| POST | `/api/trips/:tripId/votes/:voteId/answer` | Подать/сменить/снять голос |
| DELETE | `/api/trips/:tripId/votes/:voteId/answer` | Снять голос |

### 7.5. Бюджет

| Метод | Путь | Назначение |
|---|---|---|
| GET | `/api/trips/:tripId/expenses` | Список расходов с долями |
| POST | `/api/trips/:tripId/expenses` | Добавить расход (автоматическое разделение) |
| DELETE | `/api/trips/:tripId/expenses/:id` | Удалить расход (автор или владелец) |

### 7.6. Чат и ИИ

## 8. Реализация функциональных модулей

### 8.1. Аутентификация

При регистрации пароль хешируется bcrypt, создаётся пользователь, формируется JWT `{ sub, email }` и записывается в HttpOnly-cookie `fjp_session` со сроком 7 дней. Стратегия `passport-jwt` извлекает токен из cookie через собственный экстрактор. На клиенте контекст `AuthProvider` при загрузке приложения запрашивает `/users/me`: при `401` пользователь считается анонимным, иначе — авторизованным. Приватные маршруты обёрнуты компонентом `RequireAuth`, перенаправляющим неавторизованных на страницу входа.

### 8.2. Поездки и разграничение доступа

Создание поездки автоматически добавляет автора как участника с ролью `owner`. Чтение/изменение поездки проверяет членство через вспомогательный метод `TripsService.assertMember(tripId, userId)`, который используется всеми вложенными модулями (расписание, голосования, расходы, чат). Изменение и удаление самой поездки разрешено только владельцу.

### 8.3. Участники

Приглашение выполняется по email: сервер находит зарегистрированного пользователя и создаёт запись `TripMember`. Возвращаются осмысленные коды ошибок: `404` — пользователь не найден, `409` — уже участник. Удаление участника доступно владельцу; владельца удалить нельзя.

### 8.4. Расписание

Дни и пункты вложены в поездку. Пункт имеет тип (перелёт, ночёвка, еда, прогулка, переезд, место, прочее), время, заголовок, заметку и опциональные координаты с адресом. Порядок пунктов поддерживается полем `sortOrder`. На клиенте дни сворачиваются/разворачиваются, пункты добавляются инлайн-формой с поиском места.

### 8.5. Голосования

Поддерживается одиночный выбор: повторный клик по выбранному варианту снимает голос, клик по другому — переносит его (в транзакции: удаление прежних голосов пользователя и создание нового). Сервер возвращает по каждому голосованию список вариантов с количеством голосов, суммарное число голосов и идентификатор варианта текущего пользователя — клиенту не нужно ничего досчитывать.

### 8.6. Бюджет

При добавлении расхода сумма автоматически делится поровну между выбранными участниками: создаются записи `ExpenseSplit` с долей `amount / N`, округлённой до копеек. Баланс каждого участника вычисляется на клиенте: плательщику прибавляется полная сумма, каждому участнику доли — вычитается его доля. Расходы имеют категории (жильё, еда, транспорт, активности, прочее), используемые в сводке итогов.

### 8.7. Чат

Сообщения сохраняются в БД (`TripMessage`). Сообщение бота отличается флагом `isBot` и отсутствием автора. Отправка сообщения, содержащего упоминание `@гид`, инициирует обращение к ИИ: формируется контекст из последних сообщений и данных поездки, ответ модели сохраняется как сообщение бота. Сбой ИИ не нарушает чат — пользовательское сообщение сохраняется в любом случае, а ошибка возвращается отдельным полем.

> Примечание по реализации: упоминание распознаётся регулярным выражением с учётом кириллицы — `/@гид(?![а-яА-ЯёЁ])/i`, поскольку стандартная граница слова `\b` в JavaScript не работает с кириллическими символами.

Долговременная «память» ассистента и контекстные подсказки в боковой панели чата подробно описаны в разделе 9 (пп. 9.3–9.4).

## 9. Интеграция искусственного интеллекта

### 9.1. ИИ-ассистент в чате

Ассистент «Гид» отвечает на вопросы участников прямо в чате. В системный промпт включается контекст поездки (название, направление, даты, состав, расписание по дням, суммарные расходы) и последние сообщения. Это позволяет давать ответы, релевантные конкретной поездке (например, оценивать суммы в рублях с учётом уже понесённых трат).

### 9.2. Структурированный анализ переписки

Ключевое техническое решение — получение **строго структурированного** результата анализа. Вместо ненадёжного парсинга текстового ответа применяется механизм **принудительного вызова инструмента** (`tool_choice`): для модели объявляется инструмент `save_chat_analysis` с JSON-схемой результата, и модель обязана ответить вызовом этого инструмента. Аргументы вызова уже соответствуют схеме, что исключает ошибки разбора.

Схема результата включает краткую сводку и набор секций: предпочтительные направления, бюджет, даты, интересы, ограничения и зоны разногласий. Каждый пункт секции содержит текст, признак высокой уверенности, признак конфликта и оценку числа подтверждающих сообщений. Результат сохраняется в `ChatAnalysis` и отображается на вкладке «Сводка»; обнаруженные разногласия предлагается вынести на голосование.

### 9.3. Иерархическая память чата

Чтобы ассистент «помнил» весь разговор, а не только последние реплики, реализована **иерархическая (свёрточная) память**. По мере накопления переписки каждые N сообщений (параметр `CHAT_SUMMARY_BLOCK_SIZE`, по умолчанию 100) очередной блок чата сжимается в краткую структурированную «память» отдельным обращением к модели через механизм принудительного вызова инструмента `save_chat_summary`. Результат описывает блок в полях: общее настроение (`mood`), краткая сводка, ключевые темы, принятые решения и открытые вопросы; он сохраняется в таблице `ChatSummary`.

При ответе ассистента контекст формируется иерархически: все свёрнутые сводки блоков (сжатая память всего чата) + хвост последних несвёрнутых сообщений + данные поездки. Такой подход удерживает объём контекста ограниченным независимо от длины переписки, сохраняя «долговременную память» о ранних договорённостях. Генерация сводок идемпотентна: формируются только полностью завершённые и ещё не обработанные блоки. Накопленная память отображается в боковой панели чата (настроение + сводка по каждому блоку).

### 9.4. Генерация контекстных подсказок

Боковая панель чата предлагает живые подсказки по поездке, формируемые **за один запрос** к модели (инструмент `save_suggestions`). На вход подаётся полный контекст поездки (направление, даты, маршрут по дням), и модель, опираясь на свои знания о регионе, возвращает 5–6 конкретных идей: куда сходить, что попробовать, на что заложить время. Каждая подсказка содержит заголовок (по возможности привязанный к конкретному дню), краткое описание с практической конкретикой и готовый вопрос, который участник может одним кликом подставить в поле ввода и задать ассистенту для углубления.

### 9.5. Деградация при отсутствии ключа

Если ключ ИИ не сконфигурирован, соответствующие эндпоинты возвращают `503` (или пустой результат для подсказок) с понятным сообщением, а интерфейс корректно отображает подсказку по настройке — приложение остаётся работоспособным без ИИ.

## 10. Картографическая подсистема

Карта реализована на Mapbox GL JS. Компонент-обёртка управляет жизненным циклом карты (инициализация, добавление/удаление маркеров, очистка при размонтировании). Точки маршрута берутся из пунктов расписания, имеющих координаты, и окрашиваются по дням; легенда показывает реальные дни поездки с числом «помеченных/всего» точек, наведение подсвечивает точки конкретного дня.

Координаты добавляются через поиск места: при вводе названия выполняется обращение к Geocoding API Mapbox (с задержкой ввода для экономии запросов), выбор результата заполняет заголовок пункта, координаты и адрес. При отсутствии токена карта корректно деградирует, показывая подсказку по настройке.

## 11. Безопасность

- **Пароли** хранятся только в виде bcrypt-хешей.
- **Сессия** передаётся в cookie с флагами `HttpOnly` (недоступна из JavaScript, защита от XSS), `SameSite=Lax` и `Secure` (в production).
- **CORS** настроен на конкретный origin клиента с разрешением передачи учётных данных.
- **Валидация** всех входных DTO глобальным `ValidationPipe` с режимами `whitelist` и `forbidNonWhitelisted` (отбрасывание и запрет посторонних полей).
- **Разграничение доступа**: каждая операция над вложенными ресурсами поездки проверяет членство пользователя; межпоездочный доступ блокируется (`403`).
- **Серверные секреты** (строка подключения к БД, секрет JWT, ключ ИИ) хранятся в переменных окружения и не попадают в систему контроля версий.

- Node.js и npm;

- (опционально) токен Mapbox для карты;
