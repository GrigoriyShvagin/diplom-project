## РЕФЕРАТ

Отчёт содержит описание спроектированной и реализованной веб-платформы «Friends Journey», предназначенной для совместного планирования путешествий группами от 2 до 6 человек.

**Ключевые слова:** ВЕБ-ПРИЛОЖЕНИЕ, ОДНОСТРАНИЧНОЕ ПРИЛОЖЕНИЕ, КЛИЕНТ-СЕРВЕРНАЯ АРХИТЕКТУРА, REACT, NESTJS, POSTGRESQL, PRISMA, REST API, JWT, ИСКУССТВЕННЫЙ ИНТЕЛЛЕКТ, БОЛЬШИЕ ЯЗЫКОВЫЕ МОДЕЛИ, ИЕРАРХИЧЕСКАЯ ПАМЯТЬ, КАРТОГРАФИЯ.

**Объект разработки** — программная система поддержки совместного планирования группового путешествия.

**Цель работы** — спроектировать и реализовать полнофункциональную веб-платформу, объединяющую карту, расписание, бюджет, голосования и групповое обсуждение с интеграцией ИИ-ассистента.

**Результаты.** Реализована клиент-серверная система: одностраничное приложение на React/TypeScript и REST API на NestJS/TypeScript с СУБД PostgreSQL (доступ через ORM Prisma). Спроектирована нормализованная модель данных из 17 сущностей. Реализованы аутентификация на основе JWT в защищённой cookie, разграничение доступа, девять функциональных модулей, интеграция картографического сервиса Mapbox и ИИ-ассистента на базе Anthropic Claude с механизмом структурированного вывода (tool use) и иерархической памятью чата.

**Объём программного кода:** ~80 модулей исходного кода (TypeScript/TSX), разбитых на два приложения монорепозитория.

## ВВЕДЕНИЕ

Самостоятельные групповые путешествия — массовый формат отдыха, при котором планирование сопряжено с типовыми трудностями: обсуждение «расплывается» в мессенджере, договорённости теряются, выбор мест и учёт общих расходов ведутся вручную. Существующие инструменты решают эти задачи фрагментарно (карты — отдельно, разделение счетов — отдельно, чат — в стороннем приложении).

Разрабатываемая платформа объединяет в едином пространстве: карту и точки маршрута, расписание по дням, групповые голосования, учёт и автоматическое разделение расходов, групповой чат и ИИ-ассистента, который помнит весь контекст переписки и даёт рекомендации по поездке. Настоящий отчёт даёт исчерпывающее техническое описание системы: архитектуры, модели данных, структуры исходного кода (с описанием каждого файла), ключевых алгоритмов и применённых библиотек.

## 1. Анализ предметной области и постановка задачи

### 1.1. Диаграмма вариантов использования

> **Рисунок 1 — Диаграмма вариантов использования.** Диаграмма: [`docs/diagrams/01-use-case.drawio`](docs/diagrams/01-use-case.drawio) (открыть в diagrams.net / draw.io; экспорт в PNG/SVG для вставки в записку).

### 1.2. Функциональные требования

1. Аутентификация: регистрация, вход, выход; защита приватных разделов.
2. Управление поездками: CRUD, разграничение доступа по участию.
3. Участники: приглашение по email, удаление организатором.
4. Расписание: дни и пункты (тип, время, заголовок, заметка, координаты).
5. Карта: отображение точек маршрута по дням, поиск места.
6. Голосования: создание, подача/смена/снятие голоса, подсчёт.
7. Бюджет: расходы с категориями, автоматическое разделение, баланс.
8. Чат: обмен сообщениями с сохранением, обращение к ИИ через `@гид`.
9. ИИ: ответы в чате с памятью, структурированный анализ, контекстные подсказки.

### 1.3. Нефункциональные требования

Типобезопасность (TypeScript), валидация входных данных, хеширование паролей, передача сессии в защищённой cookie, разграничение доступа на уровне участия в поездке, адаптивный интерфейс, двуязычность (RU/EN).

### 3.2. Перечень таблиц и назначение

### 3.3. Нормализация и целостность

Схема находится в третьей нормальной форме: отсутствуют повторяющиеся группы, частичные и транзитивные зависимости; агрегаты (баланс участника, итоги по категориям) не хранятся, а вычисляются. Целостность обеспечена внешними ключами и каскадным удалением (`onDelete: Cascade`) для дочерних сущностей поездки. Уникальность пар гарантирована составными ограничениями: `(tripId, userId)` в `TripMember`, `(optionId, userId)` в `VoteAnswer`, `(tripId, blockNumber)` в `ChatSummary`. Для частых выборок настроены индексы, в частности `(tripId, createdAt)` в `TripMessage`. Первичные ключи — UUID.

### 6.3. Разграничение доступа

Все операции над вложенными ресурсами поездки начинаются с вызова `TripsService.assertMember(tripId, userId)`, который загружает владельца и список участников и выбрасывает `403 Forbidden`, если пользователь не является участником, либо `404 Not Found`, если поездка отсутствует. ### 6.6. Структурированный вывод LLM (tool use)

Ключевое решение для надёжного машинного разбора ответов модели — **принудительный вызов инструмента**. Вместо разбора свободного текста (ненадёжного из-за возможных markdown-обёрток и преамбул) для модели объявляется инструмент с JSON-схемой результата, и через параметр `tool_choice` модель обязывается ответить вызовом именно этого инструмента; аргументы вызова уже соответствуют схеме.

Схема обращения: сервис вызывает `messages.create` с параметрами `system`, `tools=[схема]`, `tool_choice={type:'tool', name:'...'}`; модель возвращает блок `tool_use`, поле `input` которого уже является валидным JSON по заданной схеме и используется без разбора строк.

Приём применяется в трёх местах: анализ переписки (`save_chat_analysis`), свёртка блока чата (`save_chat_summary`), генерация подсказок (`save_suggestions`).

### 6.7. Иерархическая (свёрточная) память чата

Главный алгоритм, обеспечивающий «долговременную память» ассистента при ограниченном размере контекста.

> **Рисунок 8 — Блок-схема: свёртка памяти чата (ensureSummaries).** Диаграмма: [`docs/diagrams/08-flow-chat-memory.drawio`](docs/diagrams/08-flow-chat-memory.drawio) (открыть в diagrams.net / draw.io; экспорт в PNG/SVG для вставки в записку).

При ответе ассистента контекст собирается иерархически:

Сборка контекста для ответа ассистента иерархична: «все свёртки блоков (сжатая память всего чата)» + «хвост последних несвёрнутых сообщений» + «данные поездки» → системный промпт → Claude → ответ гида.

Размер блока N задаётся параметром `CHAT_SUMMARY_BLOCK_SIZE` (по умолчанию 100). Алгоритм идемпотентен: формируются только полностью завершённые и ещё не обработанные блоки, что позволяет безопасно вызывать его при каждой отправке сообщения.

### 6.9. Контекстные подсказки

Метод `AiService.suggest` одним обращением к модели (инструмент `save_suggestions`) формирует 5–6 идей по поездке. В системный промпт включается полный контекст поездки (направление, даты, маршрут по дням); модель использует собственные знания о регионе и привязывает идеи к конкретным дням. Каждая подсказка содержит заголовок, описание с конкретикой и готовый вопрос для углубления.

## 8. Безопасность

- Пароли хранятся только в виде bcrypt-хешей.
- Сессия передаётся в cookie с флагами `HttpOnly`, `SameSite=Lax`, `Secure` (в production), что защищает от XSS-кражи токена.
- JWT подписывается секретом из переменных окружения (алгоритм на основе HMAC).
- Глобальная валидация DTO с отбрасыванием и запретом посторонних полей.
- Разграничение доступа: каждая операция над ресурсами поездки проверяет членство; межпоездочный и анонимный доступ блокируются.
- CORS ограничен origin клиента с разрешением передачи учётных данных.
- Секреты (строка подключения, секрет JWT, ключ ИИ) хранятся вне системы контроля версий.

Применён механизм structured output (tool use) для надёжного машинного разбора ответов модели и оригинальный алгоритм иерархической памяти чата, удерживающий объём контекста ограниченным при сколь угодно длинной переписке. **Направления развития:** синхронизация в реальном времени (WebSocket/SSE), полноценный справочник POI с внешними провайдерами, импорт внешних чатов, алгоритм минимизации взаиморасчётов, разделение клиентского бандла по маршрутам.
